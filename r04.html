Pare-feu FortiGate
NAT, VIP et Firewall Policy

Terminé M2L — Atelier de Professionnalisation avr. 2026

Contexte & problématique

Dans l'infrastructure simulée M2L, l'objectif était d'installer et configurer un pare-feu de niveau professionnel (Fortinet FortiGate) pour sécuriser l'accès réseau, avec segmentation WAN/LAN et redirection de flux HTTP vers un serveur web interne via IP Virtuelle (VIP). Réalisé seul sur Mac M1 via UTM (émulation QEMU x86_64), infrastructure conforme à l'Annexe II.E du référentiel BTS SIO.

Architecture mise en place

[Internet / Réseau WiFi 172.16.1.x]
            ↓
  [port1 - WAN - DHCP - 172.16.1.84]
            |
    [FortiGate VM64-KVM - UTM]
            |
  [port2 - LAN - 172.16.1.1 statique]
            ↓
  [VM Ubuntu VMware - 172.16.1.79]
         (Apache2 — serveur web M2L)

Travaux réalisés

  • 01Téléchargement de l'image officielle FortiGate VM64-KVM v7 sur support.fortinet.com. Conversion fortios.vmdk → fortios.qcow2 via qemu-img (Homebrew).
  • 02Création de la VM dans UTM (QEMU x86_64, type q35), 2 interfaces virtio-net-pci. Montage du disque qcow2.
  • 03Premier démarrage FortiGate, login admin (mot de passe vide — image KVM). Activation de la licence évaluation 15 jours.
  • 04Configuration port1 (WAN) : mode DHCP → obtention IP 172.16.1.84. Allowaccess : ping, https.
  • 05Configuration port2 (LAN) : IP statique 172.16.1.1/24. Allowaccess : ping, https.
  • 06Installation Apache2 sur VM Ubuntu (172.16.1.79) : apt install apache2 -y.
  • 07Création IP Virtuelle (VIP) dans FortiGate : Policy & Objects → Virtual IPs → Static NAT, port1, external 172.16.1.84 → map to 172.16.1.79, port forwarding TCP 8080 → 80.
  • 08Création Firewall Policy : WAN → LAN, source all, destination VIP Serveur-web-M2L, service HTTP, action ACCEPT, NAT activé.
  • 09Test depuis navigateur : http://172.16.1.84:8080 → page Apache Ubuntu affichée. ✓
